ПРОМТ-КОНТЕКСТ — M16.1 UI Layout Rework
Проект: The Turf / Передел
О чём продукт

The Turf / «Передел» — desktop single-player card/board mini-game внутри визуальной новеллы.

В матче участвуют:

1 human player;
3 local AI;
партия длится 15 раундов;
финал — Game Over.

Текущий этап: Milestone M16.1 — UI Layout Rework.

Это дополнительный UI-polish milestone после M16 и до M17.

Цель M16.1 — переделать уже рабочий M16 UI из “панельного/debug-like интерфейса” в более цельную настольную tabletop-композицию:

игровой стол;
зоны игроков;
рука human player;
рынок карт;
action/control area;
лог;
контракт/контакты;
readable desktop layout.

M16.1 не меняет gameplay rules, balance, deterministic replay, public facade contracts или state schema.

Архитектура

Проект logic-first, data-driven, deterministic.

Dependency direction:

UI -> GameStateManager -> logic -> catalogs/resources/constants

Правила:

UI является view/input layer.
Runtime state принадлежит GameStateManager.
UI вызывает gameplay только через GameStateManager.
UI не читает и не мутирует GameStateManager.state.
Failed validation не мутирует active state.
Selectors/previews не мутируют state и не consume random.
UI-local state хранит только selection/hover/focus/layout state.
Gameplay modules не зависят от UI scenes.
CardView остаётся template-based widget.
M15 replay determinism и M16 UI boundaries должны остаться зелёными.

Договорённости по коду
M16.1 не меняет gameplay rules.
Не менять IDs, formulas, balance, phase order, AI policy и winner rules.
Не менять deterministic random algorithm/call order.
Не менять state schema.
Не менять public GameStateManager contracts без подтверждённого UI blocker.
Не добавлять save/load, persistence, profiles или campaign progression.
Не добавлять new gameplay selectors молча.
Не дублировать gameplay validation в UI.
Не читать raw GameStateManager.state.
Не вызывать low-level owner modules из UI.
Не переносить owner logic в UI helpers.
CardView остаётся template-based.
UI layout может хранить только UI-local presentation state.
Каждый .gd файл строго меньше 250 строк.
Static scans нельзя ослаблять.
UI boundary tests должны оставаться зелёными.
Если найден real defect — сначала failing regression test, потом минимальный fix.
Неясность оформляется как OQ-*.
