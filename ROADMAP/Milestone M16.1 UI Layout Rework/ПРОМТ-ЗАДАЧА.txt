The Turf / «Передел» — desktop single-player card/board mini-game с одним human player и тремя local AI.

## Layout goal

Перевести GameScreen к tabletop composition.

Baseline target:

```text
GameScreen / Tabletop Board

┌─────────────────────────────────────────────────────────────┐
│ Phase / Round / Active player / Error feedback              │
├───────────────┬───────────────────────────────┬─────────────┤
│ AI Zone       │ Market / Central Table         │ AI Zone     │
│ ai_1          │ active phase content           │ ai_2        │
├───────────────┴───────────────────────────────┴─────────────┤
│ AI Zone ai_3 / shared board summaries                       │
├───────────────────────────────┬─────────────────────────────┤
│ Human board + hand/action     │ Contract/Contact/Log column │
└───────────────────────────────┴─────────────────────────────┘
```

Exact node tree may differ, but layout must achieve:

* clear center table;
* clear human interaction zone;
* clear AI opponent zones;
* market/action/street deal as central phase content;
* side info column not blocking core actions;
* logs readable but not dominant;
* player boards visible without huge scrolling at 1920×1080;
* 1280×720 usable with ScrollContainer where needed.

## Desktop layout requirements

* target 1920×1080;
* minimum 1280×720;
* landscape;
* mouse input;
* all core actions reachable by buttons;
* no required drag-and-drop;
* readable labels;
* no off-screen essential controls;
* ScrollContainer allowed for logs/side areas;
* central market/action controls must remain reachable;
* phase header always visible;
* error/disabled reason visible near relevant action area.

Use:

* Control nodes;
* Containers;
* anchors;
* size flags;
* minimum sizes;
* ScrollContainer;
* Theme overrides only if needed.

Avoid:

* absolute magic positions where Containers work better;
* hardcoded window-size assumptions;
* default debug-like vertical stack as final layout;
* nested containers that make controls unreachable;
* UI layout that only works at 1920×1080.

## Screen / panel responsibilities

### Main

* project entry scene;
* owns no gameplay state;
* instantiates GameRoot;
* keeps main theme/window settings compatible.

### GameRoot

* screen host/coordinator;
* listens to GameStateManager signals once;
* switches Setup/Game/GameOver;
* does not duplicate panel behavior;
* does not do gameplay orchestration beyond facade calls.

### GameScreen

* main tabletop composition owner;
* owns layout zones;
* receives refreshed view from GameRoot;
* routes view slices to panels;
* clears stale UI selection on phase changes;
* shows phase/round/active player/errors.

### PlayerBoard

* compact player state card/board summary;
* must fit 4 players in tabletop layout;
* shows Nal/VP, engines/status/defense summary, role/profile/strong AI if available;
* no direct gameplay mutation.

### MarketPanel

* central phase content in Market phase;
* card row/grid should read as table market;
* selection preview near market;
* Buy/Rebuild/End Market reachable.

### ActionPanel

* human hand/action area;
* selected War card, target, mode, engine target;
* Execute/Discard/End Action reachable;
* preview/disabled reason visible.

### StreetDealPanel

* central modal/table card-like offer during Street Deal phase;
* option buttons clear and reachable;
* uses canonical `option_a` / `option_b`.

### ContractPanel / ContactPanel / GameLogPanel

* side column / secondary info;
* do not dominate central gameplay;
* scroll/expand where needed;
* readable at 1280×720.

### GameOverScreen

* final result presentation;
* New Game control;
* no gameplay commands after Game Over.

## Defect fix policy

Можно исправлять только:

* confirmed UI layout defect;
* scene parse/runtime error;
* stale UI-local state defect;
* UI static boundary gap;
* UI-to-facade integration defect;
* missing presentation view needed by layout;
* test-confirmed regression.

Для каждого fix:

1. Покажи reproduction/failing test.
2. Укажи owner file/module.
3. Добавь regression test.
4. Внеси минимальное изменение.
5. Запусти targeted, affected и full suites.

Не исправляй gameplay через UI.
Не подгоняй expected result теста под bug.
