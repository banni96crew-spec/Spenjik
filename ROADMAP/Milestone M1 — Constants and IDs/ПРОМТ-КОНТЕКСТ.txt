# Проект: The Turf / Передел

## О чём продукт

“The Turf” / “Передел” — desktop single-player card/board mini-game внутри визуальной новеллы.

Проект строится по milestone-подходу: от инфраструктуры и стабильных идентификаторов к данным, gameplay logic, facade и только затем к UI.

Игровое ядро должно быть:

- детерминированным;
- тестируемым без UI;
- data-driven;
- разделённым на небольшие модули;
- защищённым от неявного изменения правил.

Основные требования находятся в `res://docs/prd/`. Coding-agent обязан использовать соответствующий owner PRD как источник истины и не должен реализовывать правила по памяти.

Если правило отсутствует, противоречиво или требует нового ID/error code, нельзя угадывать решение. Нужно остановить затронутую часть работы и зарегистрировать вопрос `OQ-*` в `res://docs/prd/21_OPEN_QUESTIONS_AND_FIXES.md`.

## Стек

- Engine: Godot 4.6.2 stable.
- Language: GDScript.
- Tests: GUT.
- Config data: `.tres` Resources, начиная с будущих milestone.
- Runtime state: Dictionary snapshots, начиная с будущих milestone.
- Gameplay facade: `GameStateManager.gd` Autoload, реализуется позже.
- Architecture: logic-first, data-driven, UI as view layer.
- Export targets: Windows и Linux в первую очередь.
- No web stack.
- No gameplay persistence в MVP.

## Архитектура

Будущее направление зависимостей:

UI → GameStateManager → logic modules → catalogs/resources/constants

Архитектурные правила:

- UI не владеет gameplay logic.
- Gameplay logic не должна находиться в сценах.
- Стабильные ID и validation error codes принадлежат `res://data/ids/`.
- Повторяемые gameplay-строки не должны быть захардкожены в logic modules, если ими владеет constants module.
- `.tres` Resources в будущем должны ссылаться на стабильные ID.
- Runtime state в будущем должен использовать те же стабильные ID.
- Failed validation в будущих mutators не должна изменять активное состояние.
- Gameplay random в будущем разрешён только через `SeededRandom.gd` и `SeededPicker.gd`.
- Каждый `.gd` файл должен содержать менее 250 строк.
- Использовать static typing там, где это практично.
- Не создавать god-object files.

## Договорённости по коду

- Использовать Godot 4.6.2 и GDScript.
- До изменений проверить M0 gate и текущее состояние репозитория.
- Сохранять существующие пользовательские изменения.
- Точные ID брать из `03_IDS_AND_CONSTANTS.md` и owner modules.
- Не переименовывать и не изменять защищённые PRD ID.
- Не добавлять ID “на будущее”, если они не определены PRD.
- Не возвращать ad-hoc error strings.
- Validation error codes должны быть стабильными константами.
- Не дублировать одно значение в разных owner-файлах.
- Использовать `StringName` или тип, прямо установленный owner PRD; не менять тип самостоятельно.
- Использовать `class_name`, если это соответствует архитектуре owner PRD.
- Использовать static typing там, где это практично.
- Каждый `.gd` файл должен быть меньше 250 строк.
- Добавлять GUT-тесты одновременно с constants.
- Тесты должны проверять наличие обязательных constants и отсутствие дубликатов.
- Не ослаблять тесты для получения формального pass.
- Не заявлять, что тесты прошли, если они не запускались.
- Не оставлять `TODO`, `TBD`, `FIXME` или `???` без ссылки на `OQ-*`.
- При неясности зарегистрировать `OQ-*`, а не придумывать значение.
