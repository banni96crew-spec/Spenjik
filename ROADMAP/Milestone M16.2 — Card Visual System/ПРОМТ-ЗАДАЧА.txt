## Контекст проекта

The Turf / «Передел» — desktop single-player card/board mini-game с одним human player и тремя local AI.

M0–M16/M16.1 считаются prerequisites, но readiness нужно проверить по repository/tests.

M16 дал functional UI.
M16.1 отвечает за tabletop layout.
M16.2 отвечает за карточную визуальную систему.

M16.2 должен сделать так, чтобы карты в UI выглядели как часть одной тёмной физической колоды, основанной на утверждённых reference-картах.

Dependency direction:

`UI -> GameStateManager -> logic modules -> catalogs/resources/constants`

M16.2 не меняет gameplay rules.
M16.2 не меняет deterministic replay.
M16.2 не добавляет persistence/save/load.
M16.2 не является M17 hardening.

## Visual style lock

Общий стиль:

* vertical dark tabletop card;
* black worn matte card stock;
* rounded corners;
* thin distressed warm off-white border;
* criminal underground board-game mood;
* high-resolution pixel-art engraving / pixelated ink linework direction;
* dirty off-white ink on black;
* dense stippling/hatching feeling where practical in UI;
* worn print texture;
* limited palette:

  * black;
  * dirty ivory/off-white;
  * muted gray/brown grime;
  * tiny dark red accent only where appropriate;
* no glossy mobile UI;
* no bright fantasy colors;
* no anime;
* no cyberpunk neon overload;
* no clean vector/mobile game look;
* no smooth 3D render.

Composition baseline:

* price in top-left;
* small custom currency glyph near price;
* card type marker under/near price;
* centered art/illustration zone;
* lower title panel;
* large distressed title;
* readable rules/effect text below title;
* small type icon in lower-right or equivalent reference-consistent marker;
* thin divider between illustration and title/effect area.

---

## Card type visual distinction

Engine:

* gear marker;
* machinery/economy feeling;
* dirty industrial visual language;
* gear icon reserved for Engine only.

Status:

* status/crown-style marker if approved by reference;
* stash/influence/property feeling;
* do not reuse gear;
* do not reuse attack/shield marker.

War:

* aggressive attack marker;
* sharp/threatening visual language;
* attack marker reserved for War only;
* no shield/gear confusion.

Defense:

* shield/protection marker;
* defensive/security feeling;
* shield marker reserved for Defense only;
* do not reuse gear or attack marker.

Important:

* visual symbols are presentation only;
* do not invent new gameplay meaning through icons;
* if exact marker from reference is unclear, preserve type separation and document chosen marker.

## CardView responsibilities

CardView must display:

* title;
* type;
* price;
* effect/rules text as display text only;
* optional card id/debug label only if already present and not visually dominant;
* type marker;
* selected/hover state;
* disabled/unavailable state;
* affordable/unaffordable state if supplied by view/panel;
* card slot/empty state if used by panel.

CardView must not:

* compute price;
* compute affordability;
* validate purchase/attack;
* parse effect text as rules;
* call MarketLogic, PriceLogic, CombatEngine or any owner module;
* read GameStateManager.state;
* mutate gameplay state;
* consume random;
* special-case specific card IDs.

---

## Visual states

Implement or preserve clear presentation states:

Normal:

* readable card;
* type marker visible;
* price/title/effect visible.

Hover:

* subtle highlight;
* no gameplay mutation;
* mouse feedback only.

Selected:

* stronger outline/glow/raised state;
* must remain readable.

Disabled/unavailable:

* dimmed or desaturated;
* effect/title still readable enough;
* no fake “broken card” gameplay meaning.

Affordable/unaffordable:

* if panel supplies it, price zone can visually indicate affordability;
* do not compute affordability inside CardView.

Owned/placed/hand/market context:

* if context supplied, CardView may adjust scale/layout labels;
* no gameplay logic.

---

## Layout requirements inside card

CardView should have stable internal zones:

* outer worn border;
* top price/type zone;
* central art/illustration placeholder zone;
* lower title panel;
* lower effect text zone;
* optional bottom-right type marker.

Use Control/Container nodes where practical.

Do not create a single flat TextureRect card image.

Allowed:

* StyleBoxFlat/StyleBoxTexture if project assets exist;
* Label theme overrides;
* simple procedural panel styling;
* type marker labels/icons;
* placeholder illustration blocks;
* custom font/theme only if already available or safely added.

Avoid:

* hardcoded absolute positioning that breaks scaling;
* tiny unreadable labels;
* giant cards that break 1280×720;
* default blue/gray Godot look;
* full-card PNG as sole render path.

## Defect fix policy

Можно исправлять только:

* CardView visual system defect;
* scene parse/runtime error;
* CardView scaling/layout issue;
* missing presentation field needed for visual rendering;
* UI static boundary gap;
* test-confirmed regression.

Для каждого fix:

1. Покажи reproduction/failing test.
2. Укажи owner file/module.
3. Добавь regression test.
4. Внеси минимальное изменение.
5. Запусти targeted, affected и full suites.

Не исправляй gameplay через CardView.
Не подгоняй expected result теста под bug.
