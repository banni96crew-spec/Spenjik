## Контекст проекта

The Turf / «Передел» — desktop single-player card/board mini-game с одним human player и тремя local AI.

M0–M16.3 считаются prerequisites, но readiness нужно проверить по repository/tests.

M16 дал functional UI.
M16.1 отвечает за layout.
M16.2 отвечает за Card Visual System.
M16.3 отвечает за Tabletop Atmosphere.
M16.4 отвечает за UI animation and feedback.

Dependency direction:

```text
UI -> GameStateManager -> logic modules -> catalogs/resources/constants
```

M16.4 не меняет gameplay rules.
M16.4 не меняет deterministic replay.
M16.4 не добавляет persistence/save/load.
M16.4 не является M17 hardening.

## Feedback style direction

Feedback должен ощущаться как:

* тяжёлый tabletop;
* физические карты;
* короткие движения;
* грязный криминальный стол;
* understated but satisfying;
* readable action confirmation.

Feedback не должен ощущаться как:

* mobile game juice spam;
* bright arcade effects;
* magic particles;
* cyberpunk neon flash;
* fantasy spell VFX;
* animation that delays or hides gameplay result too long.

Preferred feel:

* card lift on hover;
* selected card slight raise/outline;
* failed click shake/red pulse;
* purchase small card pulse/place feedback;
* attack quick impact flash;
* block shield/cops/cartel/judge flash;
* phase transition short fade/slide;
* Game Over slow reveal.

## Animation ownership rules

M16.4 may:

* add Tween animations to UI nodes;
* animate `modulate`, `scale`, `position`, `rotation`, `self_modulate`, `theme_override` where safe;
* add UI-only feedback layer;
* add presentation-only signals between UI widgets and parent panels;
* add short presentation lock to prevent double-click spam;
* animate GameOver reveal;
* animate hover/selected/disabled states;
* animate action_failed feedback.

M16.4 may not:

* change gameplay state;
* call owner modules directly;
* read `GameStateManager.state`;
* consume random;
* use physics for UI feedback;
* add gameplay delays to state transition;
* make gameplay success depend on animation completion;
* use wall-clock timestamps in gameplay state/log;
* add audio;
* add save/load;
* change replay scripts.

## Recommended feedback map

Card hover:

* CardView slightly raises/scales.
* No gameplay call.
* No selection mutation unless click.

Card selected:

* selected outline/glow/raise.
* selected state remains UI-local.

Disabled card/action:

* dim/pulse/shake on attempted action.
* show DisabledReasonLabel.

Purchase success:

* card pulse/place flash in MarketPanel.
* update happens from refreshed GameStateManager view.
* no fake purchase animation if facade result failed.

Attack success:

* quick impact flash on ActionPanel/target area.
* optional log highlight.
* no combat result recomputation.

Attack blocked:

* brief shield/defense badge flash.
* use result/log/view data only.
* no block calculation in UI.

Discard:

* card fade/slide feedback.
* state update still comes from facade.

Phase transition:

* short fade/slide between phase content.
* header phase highlight.
* must not hide essential errors.

Street Deal choice:

* selected option pulse.
* panel refresh after facade result.

Contact unlock:

* contact card/option pulse.
* no fake unlock if facade result failed.

Contract claim:

* claim success pulse.
* no reward calculation in UI.

Denied action:

* short shake/pulse + readable reason.
* no state mutation.

Game Over:

* slow reveal of winner/result.
* New Game remains accessible.

## Timing constraints

Animations should be short:

* hover: instant to 0.12s;
* selected: 0.08–0.15s;
* denied shake/pulse: 0.15–0.25s;
* purchase/attack feedback: 0.15–0.35s;
* phase transition: 0.2–0.45s;
* Game Over reveal: 0.5–1.0s.

Avoid long blocking animations.

No gameplay command should require waiting for a long animation to complete.

---

## Presentation lock

If needed, implement UI-local presentation lock:

* prevents repeated clicks while a command is being processed/feedback starts;
* clears after short animation or immediately after state refresh;
* does not write gameplay state;
* does not block GameStateManager internally;
* does not affect replay tests;
* can be skipped in tests if difficult, but must not cause double-submit bugs.

Do not add global gameplay lock.

## Tests

Add/update focused tests.

Animation/feedback tests should verify:

* CardView hover/selected methods exist or signals can trigger feedback.
* CardView selected/disabled state is UI-local.
* Feedback methods do not call GameStateManager directly unless they are panel command handlers.
* Denied feedback can be triggered without state mutation.
* Action failed signal produces visible/error feedback.
* Phase transition feedback can run without changing state.
* GameOver reveal scene instantiates.
* UI feedback scripts contain no owner module calls.
* UI feedback scripts contain no gameplay random.
* All animation scripts stay under 250 lines.

Static tests should verify:

* no `GameStateManager.state`;
* no owner module calls;
* no SeededRandom/SeededPicker;
* no randf/randi/randomize/RandomNumberGenerator;
* no FileAccess/user persistence;
* no audio nodes/files in M16.4 scope;
* no hardcoded gameplay effect calculations.

Affected existing tests must remain green:

* UI payload tests;
* UI layout tests;
* CardView visual tests;
* atmosphere tests;
* canonical smoke;
* integration/replay;
* full suite.

Do not use replay tests to assert animation timing.
Do not weaken existing tests.

## Defect fix policy

Можно исправлять только:

* animation/feedback defect;
* scene parse/runtime error;
* duplicate click UI issue;
* missing UI-local state reset;
* signal connection bug;
* UI static boundary gap;
* test-confirmed regression.

Для каждого fix:

1. Покажи reproduction/failing test.
2. Укажи owner file/module.
3. Добавь regression test.
4. Внеси минимальное изменение.
5. Запусти targeted, affected и full suites.

Не исправляй gameplay через animation.
Не подгоняй expected result теста под bug.
