# Проект: The Turf / Передел

## О чём продукт

“The Turf” / “Передел” — desktop single-player card/board mini-game внутри визуальной новеллы.

Игровая система должна быть реализована как отдельное детерминированное и тестируемое ядро. Интерфейс в будущих milestone служит только слоем отображения и ввода, но не владельцем игровых правил.

Основные источники требований находятся в `res://docs/prd/`. При конфликте или недостатке информации нельзя придумывать механику: необходимо остановить затронутую часть работы и зарегистрировать вопрос формата `OQ-*` в `res://docs/prd/21_OPEN_QUESTIONS_AND_FIXES.md`.

## Стек

- Engine: Godot 4.6.2 stable.
- Language: GDScript.
- Tests: GUT.
- Config data: `.tres` Resources в будущих milestone.
- Runtime state: Dictionary snapshots в будущих milestone.
- Export targets: Windows и Linux в первую очередь.
- Persistence: отсутствует в MVP; допустим только необязательный debug snapshot в будущих milestone.
- Проект не использует web stack.

## Архитектура

Основное направление зависимостей будущей системы:

UI → GameStateManager → logic modules → catalogs/resources/constants

Архитектурные принципы:

- logic-first;
- data-driven;
- UI только отображает состояние и отправляет команды;
- игровые данные в будущем хранятся в `.tres`;
- runtime state в будущем хранится в Dictionary snapshots;
- `res://autoload/GameStateManager.gd` в будущем будет единственным gameplay facade для UI;
- gameplay logic не должна находиться в сценах и UI-скриптах;
- игровые Resources нельзя изменять во время выполнения;
- неуспешная валидация не должна изменять активное состояние;
- selectors и previews не должны изменять состояние или потреблять random;
- каждый `.gd` файл должен содержать менее 250 строк.

`GameStateManager.gd`, Resources, runtime state и gameplay modules не реализуются в M0.

## Договорённости по коду

- Использовать только Godot 4.6.2 и GDScript.
- Сначала изучить актуальное состояние репозитория и связанные PRD.
- Не перезаписывать и не удалять существующие пользовательские изменения без необходимости.
- Не создавать пустую имитацию GUT: `res://addons/gut/` должен содержать реально подключённый addon.
- Bootstrap smoke должен находиться в `res://tests/smoke/test_gut_bootstrap.gd` и быть обычным минимальным GUT-тестом без gameplay logic.
- Канонический MVP smoke `res://tests/integration/test_smoke_mvp.gd` не входит в M0: он требует интегрированных gameplay-систем и становится обязательным не позднее M15.
- Не заявлять, что проект или тесты работают, если они фактически не запускались.
- Все создаваемые `.gd` файлы должны быть меньше 250 строк.
- Не использовать непроверенные `TODO`, `TBD`, `FIXME` или `???`. Неясность должна быть связана с `OQ-*`.
- Не изменять игровые правила и значения PRD.
- Не добавлять лишние зависимости и подсистемы.
- Пути проекта указывать в формате `res://...`.
