КОНТЕКСТ ПРОЕКТА:

“The Turf” / “Передел” — desktop single-player card/board mini-game внутри визуальной новеллы.

Целевой стек:

- Godot 4.6.2 stable;
- GDScript;
- GUT;
- `.tres` Resources в будущих milestone;
- Dictionary state snapshots в будущих milestone;
- `GameStateManager.gd` как будущий Autoload facade;
- Windows и Linux как первые export targets;
- без web stack.

Архитектура проекта — logic-first и data-driven. UI в будущем является только view/controller layer и не владеет gameplay logic.

Текущий milestone не реализует игровую архитектуру целиком. Он создаёт только рабочую основу проекта, структуру каталогов и тестовую инфраструктуру.

ЗАДАЧА:

Реализуй `Milestone M0 — Project Bootstrap`.

Необходимо:

1. Проверить текущее состояние репозитория и сохранить существующие пользовательские изменения.
2. Создать корректный проект Godot 4.6.2.
3. Создать folder structure из `res://docs/prd/15_GODOT_ARCHITECTURE.md`.
4. Подключить реальный addon GUT, совместимый с Godot 4.6.2.
5. Включить GUT как plugin, если это требуется выбранной версией addon.
6. Создать пустые каталоги для unit, fixture, smoke и static tests.
7. Создать `res://tests/smoke/test_gut_bootstrap.gd` — минимальный infrastructure-only GUT bootstrap smoke, подтверждающий, что test runner работает.
8. Создать shell-helper для запуска static scans.
9. Не реализовывать сами будущие static scans за пределами минимальной bootstrap-оболочки M0.
10. Проверить открытие проекта через Godot 4.6.2.
11. Фактически запустить smoke test и сообщить точный результат.
12. Проверить отсутствие gameplay logic и запрещённых web-stack файлов.

Если каталоги нельзя сохранить в системе контроля версий пустыми, разрешается использовать нейтральные placeholder-файлы, например `.gitkeep`. Они не должны содержать gameplay assumptions.

ОГРАНИЧЕНИЯ:

- Не выходить за пределы M0.
- Не реализовывать gameplay logic.
- Не создавать карты, роли, контракты, контакты, Street Deals, долги, AI, economy, market, combat или Turf Levels.
- Не создавать Resource schemas, catalogs или игровые `.tres`.
- Не реализовывать `GameStateManager.gd`.
- Не регистрировать gameplay Autoload.
- Не создавать runtime state или игровые Dictionary schemas.
- Не реализовывать deterministic random.
- Не создавать игровые сцены и UI-компоненты из будущих milestone.
- Допустим только минимальный bootstrap placeholder, если без него нельзя проверить запуск проекта. Он не должен содержать gameplay logic.
- Не менять игровые правила, баланс, ID, API или state schemas из PRD.
- Не добавлять React, TypeScript, JavaScript gameplay runtime, Tailwind, Zustand, Docker, backend, web routing или browser-first architecture.
- Не использовать `randf()`, `randi()`, `randomize()` и `RandomNumberGenerator` в gameplay logic.
- Каждый `.gd` файл должен быть меньше 250 строк.
- Не создавать фиктивную пустую папку вместо настоящего GUT addon.
- Не ослаблять smoke test ради получения формального pass.
- Не создавать в M0 канонический MVP smoke `res://tests/integration/test_smoke_mvp.gd`: его gameplay-зависимости появляются только в последующих milestone, а сам тест обязателен не позднее M15.
- Не заявлять о прохождении проверки, если команда не была запущена.
- Не удалять и не откатывать несвязанные изменения пользователя.
- Не оставлять `TODO`, `TBD`, `FIXME` или `???` без ссылки на `OQ-*`.
- Не придумывать отсутствующие требования.

Если информации недостаточно:

1. Останови только затронутую часть реализации.
2. Проверь существующие `OQ-*` и `FIX-*`.
3. При необходимости добавь новый `OQ-*` в `res://docs/prd/21_OPEN_QUESTIONS_AND_FIXES.md`.
4. Не подменяй вопрос собственной архитектурной или gameplay-догадкой.
5. Продолжай только те части M0, которые однозначно определены.
