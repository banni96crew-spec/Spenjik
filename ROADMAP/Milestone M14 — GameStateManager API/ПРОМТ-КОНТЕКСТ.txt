# Проект: The Turf / Передел

## О чём продукт

The Turf / «Передел» — desktop single-player card/board mini-game
внутри визуальной новеллы.

В матче участвуют один человек и три локальных AI.
Игровая партия длится до 15 раундов и использует детерминированный state.

Текущий этап: Milestone M14 — GameStateManager API.

## Стек

- Godot 4.6.2 stable.
- GDScript со статической типизацией, где это практично.
- GUT.
- `.tres` Resources для статических definitions.
- Dictionary snapshots для runtime state.
- Windows/Linux first.
- Без web stack.

## Архитектура

- Logic-first, data-driven.
- `GameStateManager.gd` — единственный gameplay facade для UI.
- Он является Autoload с именем `GameStateManager`.
- Он владеет единственным активным runtime snapshot `state`.
- Gameplay rules остаются в owner logic modules M5–M13.
- Facade принимает commands, создаёт deep working copy и делегирует.
- Active state заменяется только после успешного result и validation.
- Failed mutator не изменяет active state ни частично, ни полностью.
- Selectors и previews не мутируют state и не consume random.
- UI не получает writable reference на active state.
- UI не вызывает low-level gameplay mutators напрямую.
- AI использует те же owner modules и validation boundaries.
- Gameplay random существует только в `state["random"]`.
- Resources не мутируют во время run.
- GameStateManager не является местом для gameplay formulas.

## Договорённости по коду

- `GameStateManager.gd` строго меньше 250 строк.
- Каждый другой `.gd` файл также меньше 250 строк.
- Public API следует `16_GAME_STATE_MANAGER_API.md`.
- Используется `advance_phase()`, не устаревший `next_phase()`.
- Dedicated rebuild API не маскируется fake card ID.
- Stable errors берутся из `ValidationErrors.gd`.
- Mutators возвращают stable structured results.
- Selectors возвращают deep copies или safe views.
- Signal payload не раскрывает writable active state.
- Final returned state проверяется GameStateValidator до commit.
- Random не вызывается и не реализуется внутри facade.
- Setup сохраняет установленный M0–M13 random call order.
- Изменения owner modules допускаются только для API compatibility.
- Нерешённое расхождение фиксируется как `OQ-*`.
- Ориентиры объёма промтов приблизительные, не жёсткие.
