# Проект: The Turf / Передел

## О чём продукт

“The Turf” / “Передел” — desktop single-player card/board mini-game внутри визуальной новеллы.

Проект реализуется последовательно по milestone-подходу:

M0 Project Bootstrap
→ M1 Constants and IDs
→ M2 Resource Schemas and Catalogs
→ последующие gameplay milestone

Игровая архитектура должна быть:

- logic-first;
- data-driven;
- детерминированной;
- тестируемой без UI;
- разделённой на небольшие модули;
- защищённой от неявного изменения правил.

Документы в `res://docs/prd/` являются источниками истины. Coding-agent не должен дополнять отсутствующие правила собственными предположениями.

При недостатке или конфликте информации необходимо остановить затронутую часть работы и зарегистрировать вопрос `OQ-*` в:

`res://docs/prd/21_OPEN_QUESTIONS_AND_FIXES.md`

## Стек

- Engine: Godot 4.6.2 stable.
- Language: GDScript.
- Tests: GUT.
- Static gameplay data: `.tres` Resources.
- Resource access: domain catalogs.
- Future runtime state: Dictionary snapshots.
- Future gameplay facade: `GameStateManager.gd` Autoload.
- Architecture: logic-first, data-driven, UI as view layer.
- Export targets: Windows и Linux.
- No web stack.
- No gameplay persistence в MVP.

## Архитектура

Будущее направление зависимостей:

UI → GameStateManager → logic modules → catalogs/resources/constants

Правила Resource layer:

- `.tres` Resources хранят только статические gameplay-данные.
- Resources не хранят runtime state.
- Resources нельзя изменять во время выполнения.
- Logic modules не должны загружать `.tres` напрямую в разных местах.
- Доступ к Resources выполняется через domain catalogs.
- Каждый Resource должен иметь стабильный ID из `res://data/ids/`.
- ID внутри Resource должен точно совпадать с owner constants module.
- Описания и summaries являются display text и не должны парситься как gameplay logic.
- Catalog возвращает Resource по ID и предоставляет полный набор Resources своего домена.
- Catalog не владеет runtime state и не изменяет Resources.
- Отсутствующий ID должен возвращать безопасный результат, установленный PRD, без runtime fallback.
- Все `.gd` файлы должны содержать менее 250 строк.
- Использовать static typing там, где это практично.

## Договорённости по коду

- Использовать Godot 4.6.2 и GDScript.
- До изменений проверить M0 и M1 gates.
- Сохранять пользовательские изменения.
- Каждый Resource schema должен расширять `Resource`.
- Использовать `class_name`, если это предусмотрено архитектурой.
- Использовать `@export` и точные типы для сериализуемых полей.
- Использовать constants из M1 для проверки ID.
- Не дублировать constants внутри Resource schemas или catalogs.
- Не использовать description или summary как источник gameplay rules.
- Не добавлять Resource fields, отсутствующие в owner PRD.
- Не добавлять скрытые runtime fields.
- Не мутировать загруженные Resources.
- Не использовать ad-hoc загрузку Resources вне catalogs.
- Не создавать универсальный mega-catalog вместо domain catalogs.
- Каждый `.gd` файл должен быть меньше 250 строк.
- Добавлять тесты одновременно с Resource layer.
- Не ослаблять тесты ради формального pass.
- Не заявлять о прохождении тестов без фактического запуска.
- Не оставлять `TODO`, `TBD`, `FIXME` или `???` без `OQ-*`.
- При неясности создать `OQ-*`, а не придумывать schema field или значение.
