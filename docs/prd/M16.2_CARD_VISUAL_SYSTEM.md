## 1. Purpose

M16.2 turns the existing functional card widget into a reusable visual system for all gameplay cards.

The UI should display cards as part of one dark physical deck, following the approved card references, while preserving the existing gameplay architecture.

M16.2 exists to:

* make `CardView` visually stronger and less prototype-like;
* support distinct visual variants for Engine, Status, War and Defense;
* preserve template-based rendering instead of full-card PNG rendering;
* display price, type, title and effect text in stable visual zones;
* show card state clearly: normal, hover, selected, disabled and unavailable;
* keep all card visuals data-driven by card type and presentation data;
* avoid hardcoded special cases for individual cards;
* preserve `GameStateManager` as the only gameplay-facing UI API;
* preserve deterministic replay and all existing gameplay tests.

M16.2 must not implement new gameplay rules.

M16.2 does not generate all final unique card illustrations. It creates the UI system that can display cards consistently.

M16.2 may store temporary card presentation state only.

* hovered;
* selected;
* disabled visual state;
* unavailable visual state;
* focused by keyboard/mouse if implemented;
* visual context, such as market/hand/board if supplied by parent UI;

Forbidden UI-local gameplay state:

* player hand contents as authority;
* market contents as authority;
* card ownership as authority;

* combat result;
* random step;

```text

GameStateManager public selectors

```

```text

```

## 6. Card Visual Style Rules

### 6.1. Deck Consistency Rule

All cards must look like they belong to one physical deck.

Common required traits:

* vertical card shape;
* dark card body;
* worn or distressed border;
* off-white/warm muted text;
* compact price area;
* visible type marker;
* central illustration or placeholder zone;
* lower title/effect area;
* readable title;
* readable effect text;
* consistent spacing;
* consistent selected/disabled treatment.

### 6.2. Dark Criminal Card Rule

Cards should match the dark tabletop criminal game direction.

Preferred style:

* black or near-black card surface;
* dirty ivory/off-white text and border;
* muted grime tones;
* dark red accent only where appropriate;
* worn printed look;
* understated physical card feel.

Forbidden style direction:

* default Godot panel look;
* glossy mobile card UI;
* bright fantasy colors;
* anime style;
* neon cyberpunk overload;
* smooth 3D render look;
* clean vector-only look with no worn card feel.

### 6.3. Readability Rule

Visual style must not make gameplay text unreadable.

Required readable fields:

* price;
* title;
* type marker;
* effect text;
* disabled/unavailable indicator when present.

At minimum supported resolution `1280x720`, card text may be compact but must remain usable.

### 6.4. No Full-card PNG-only Rule

CardView must not use full-card PNG images as the primary card rendering system.

Forbidden primary implementation:

```text
TextureRect displays one complete card image and CardView does nothing else.
```

Allowed:

* optional illustration/placeholder inside the art zone;
* project-local textures for borders/noise/background;
* StyleBoxTexture or StyleBoxFlat for card surfaces;
* type marker assets;
* template-based node hierarchy.

### 6.5. No Specific Card Special-case Rule

CardView must not contain hardcoded special cases for individual card IDs.

Forbidden hardcoded visual branches:

```text
laundry
stash
thug
cops
```

Visual differences must be based on card type, display state or provided presentation fields.

## 7. Card Type Variants

### 7.1. Engine Variant

Engine cards represent machinery, economy and persistent infrastructure.

Engine visual traits:

* gear marker;
* machinery/economy mood;
* industrial feel;
* neutral dirty off-white styling;
* stable, useful, mechanical feeling.

Gear marker is reserved for Engine cards.

Engine marker must not be reused for Status, War or Defense cards.

### 7.2. Status Variant

Status cards represent holdings, influence and ongoing status assets.

Status visual traits:

* status/crown-style marker if approved by reference;
* property/influence mood;
* stable but less mechanical than Engine;
* no gear marker;
* no attack marker;
* no shield marker.

If exact reference marker is unclear, use a distinct presentation-only marker and document it in the implementation report.

### 7.3. War Variant

War cards represent attacks and hostile actions.

War visual traits:

* aggressive attack marker;
* sharper visual language;
* threatening tone;
* controlled red accent allowed;
* no gear marker;
* no shield marker.

Attack marker is reserved for War cards.

### 7.4. Defense Variant

Defense cards represent protection and blocking.

Defense visual traits:

* shield/protection marker;
* defensive/security mood;
* sturdy visual treatment;
* no gear marker;
* no attack marker.

Shield marker is reserved for Defense cards.

### 7.5. Marker Meaning Rule

All type markers are presentation-only.

Type markers must not introduce new gameplay meaning.

The UI must not use marker choice to determine gameplay behavior.

Gameplay behavior remains owned by card definitions and owner modules.

## 8. CardView Structure

### 8.1. Required Visual Zones

CardView must provide stable internal zones:

* outer card surface;
* border;
* top price/type zone;
* central art or placeholder zone;
* title zone;
* effect text zone;
* optional state/disabled label;
* select/action button area if used by parent panel.

Recommended structure:

```text
CardView
└── CardSurface
    ├── TopBar
    │   ├── PriceLabel
    │   └── TypeMarker
    ├── ArtFrame
    │   └── ArtPlaceholder / Illustration
    ├── Divider
    ├── TitleLabel
    ├── EffectLabel
    ├── StateLabel
    └── SelectButton
```

Exact scene tree may differ if tests and usability remain correct.

### 8.2. Required Display Fields

CardView should support display dictionaries with:

* `id`
* `title`
* `type`
* `base_price` or `price`
* `effect_summary`
* optional `disabled_reason`
* optional `selected`
* optional `disabled`
* optional `affordable`
* optional `context`
* optional `illustration_key` or presentation-only placeholder field

CardView must handle missing optional fields safely.

CardView must not crash if optional fields are absent.

### 8.3. Display Text Rule

`effect_summary` may be displayed as text.

CardView must not parse `effect_summary` as gameplay rules.

Forbidden examples:

* extracting Nal gain from text;
* detecting card type from title/effect text;
* applying combat behavior from displayed text;
* deriving price modifiers from text.

### 8.4. Price Display Rule

CardView may display a price supplied by the presentation layer or parent panel.

CardView must not compute final price.

Allowed:

* display `price`;
* display `base_price`;
* display parent-supplied preview price;
* show parent-supplied affordable/unaffordable state.

Forbidden:

* calling `PriceLogic`;
* calculating modifier order;
* checking player Nal;
* checking protected Nal.

### 8.5. Context Rule

CardView may adjust visual scale or secondary labels by context if parent supplies context.

Allowed contexts:

* market;
* hand;
* board;
* preview;
* compact;
* disabled.

Context must remain presentation-only.

## 9. Visual States

### 9.1. Normal State

Normal card state must show:

* price;
* type marker;
* title;
* effect text;
* art/placeholder zone.

### 9.2. Hover State

Hover state may show:

* slight highlight;
* subtle border change;
* cursor feedback;
* slight lift if implemented without M16.4 animation scope.

Hover must not call gameplay mutators.

Hover may request previews only where already allowed by `17_UI_UX_SPEC.md`.

### 9.3. Selected State

Selected state should be visually stronger than hover.

Allowed selected indicators:

* stronger border;
* selected outline;
* slight raised style;
* selected label;
* type-consistent accent.

Selected state is UI-local unless parent panel commits a gameplay command through `GameStateManager`.

### 9.4. Disabled / Unavailable State

Disabled state should communicate that the card cannot be used or selected.

Allowed indicators:

* dimmed card body;
* muted text;
* disabled label;
* reduced contrast while preserving readability.

Disabled state must not hide core card identity.

### 9.5. Affordable / Unaffordable State

If parent panel supplies affordability or disabled reason, CardView may show it.

Allowed:

* price color variant;
* small unavailable marker;
* dimmed select button.

Forbidden:

* computing affordability inside CardView;
* checking player Nal;
* calling validation logic.

## 10. Panel Integration Rules

### 10.1. MarketPanel

MarketPanel may use CardView for market cards.

MarketPanel owns:

* selected market card ID;
* requesting price preview through facade;
* requesting disabled reason through facade;
* calling buy/rebuild/end market through facade.

CardView must only display supplied card data and emit selection.

### 10.2. ActionPanel

ActionPanel may use CardView for hand/War cards.

ActionPanel owns:

* selected War card ID;
* selected target;
* selected mode;
* selected engine target;
* preview request through facade;
* execute/discard/end action commands through facade.

CardView must not build attack payloads.

### 10.3. PlayerBoard

PlayerBoard may display compact card summaries.

If compact cards are shown, CardView may use compact context.

PlayerBoard must not compute card effects.

### 10.4. StreetDeal / Contract / Contact Panels

These panels may use card-like presentation if already supported, but M16.2 does not own their gameplay content.

If they reuse CardView-like visuals, that usage must remain presentation-only.

## 11. Presentation Data Rules

### 11.1. Safe Dictionary Rule

Card display data must be plain Dictionary data.

Do not expose live Resource references to UI widgets.

Allowed display fields:

* IDs;
* titles;
* type strings;
* price integers;
* effect text;
* display description;
* marker key;
* visual state flags.

### 11.2. PresentationViewBuilder Extension Rule

If CardView needs missing display fields, add them through presentation/view builders in a backward-compatible way.

Allowed:

* adding display title;
* adding display type;
* adding base price;
* adding effect text;
* adding presentation-only marker key.

Forbidden:

* adding gameplay formulas;
* adding mutable Resource references;
* adding owner result objects;
* changing existing public result shape incompatibly.

### 11.3. Resource Immutability Rule

Card visuals must not mutate `.tres` Resources.

Card definitions remain immutable runtime data.
