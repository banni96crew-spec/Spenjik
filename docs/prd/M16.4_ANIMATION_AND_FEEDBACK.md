## 1. Purpose

M16.4 adds UI-only animation and immediate visual feedback to the existing tabletop UI.

The UI should feel more tactile and responsive without changing the game rules.

M16.4 exists to:

* make card hover and selected states feel physical;
* make denied actions visibly clear;
* make successful purchases visible;
* make attack, block and discard outcomes readable;
* make phase changes easier to perceive;
* make Street Deal, contact and contract actions visibly acknowledged where safe;
* make Game Over reveal feel intentional;
* prevent repeated accidental clicks through UI-local presentation lock where needed;
* preserve `GameStateManager` as the only gameplay-facing UI API;
* preserve deterministic replay and all existing gameplay tests.

M16.4 must not implement new gameplay rules.

M16.4 must not add audio. Audio belongs to M16.5.

## 3. Core Terms

| Term                      | Meaning                                                                                         |
| ------------------------- | ----------------------------------------------------------------------------------------------- |
| Feedback                  | UI-visible confirmation or denial after player action, facade signal, or view refresh.          |
| Animation                 | Presentation-only visual transition using Control properties or Tween.                          |
| Presentation Lock         | UI-local short lock that prevents duplicate clicks while a command/feedback is being processed. |
| Hover Feedback            | Visual response when the mouse is over a card/button/panel.                                     |
| Selected Feedback         | Visual response for a UI-selected card/target/option.                                           |
| Denied Feedback           | Visual response after failed/disabled command attempt.                                          |
| Phase Transition Feedback | Short visual change after `phase_changed` or equivalent view refresh.                           |
| Feedback Layer            | Optional UI-only node that displays flashes, pulses, overlays or transient labels.              |
| Gameplay State            | Authoritative runtime state owned by `GameStateManager`.                                        |

## 4. Runtime State

### 4.1. Animation State Rule

M16.4 may store temporary UI-only animation state.

Allowed UI-local state:

* is feedback currently playing;
* is presentation lock active;
* hovered card ID;
* selected card ID already allowed by UI specs;
* last visible error code;
* last visible phase ID;
* last highlighted panel;
* temporary feedback label text;
* active tween reference.

Forbidden gameplay state:

* player Nal;
* player VP;
* player hand contents as authority;
* market contents as authority;
* card ownership as authority;
* contract progress;
* contact ownership;
* debt state;
* random step;
* phase state as authority;
* combat result as authority;
* AI decision state.

Gameplay state must come from:

```text
GameStateManager.get_view()
GameStateManager public selectors
safe presentation dictionaries
facade command results
facade signals
```

UI must not read:

```text
GameStateManager.state
```

## 5. Feedback Direction

### 5.1. High-level Feel

Feedback should feel like:

* physical cards on a dark table;
* short tactile response;
* heavy criminal tabletop mood;
* clear confirmation of action;
* restrained but satisfying movement;
* quiet pressure instead of arcade effects.

Feedback should not feel like:

* mobile-game juice spam;
* bright arcade effects;
* fantasy magic particles;
* neon cyberpunk flashes;
* long unskippable animation;
* gameplay hidden behind animation.

### 5.2. Mafia Tabletop Feedback

The game is set around criminal power, debts, protection and dirty money.

Feedback should support:

* sharp denied-action shake;
* muted red danger/error pulse;
* heavy card lift/settle;
* quick impact flash for attack;
* shield/defense badge pulse for blocked attacks;
* subtle phase fade or slide;
* final result reveal like a stamped case file or table verdict.

Avoid:

* comedic bounce;
* playful cartoon squash;
* excessive particles;
* horror gore;
* bright casino slot-machine effects.

* add gameplay delays to state transition rules;
* make gameplay success depend on animation completion;
* use wall-clock time in gameplay state/log;
* add audio;
* add save/load;
* change replay scripts.

## 7. Required Feedback Map

### 7.1. Card Hover

Card hover should provide a light response.

Allowed:

* slight lift;
* small scale change;
* border highlight;
* cursor feedback.

Forbidden:

* gameplay command call on hover;
* gameplay mutation on hover;
* owner module call on hover.

### 7.2. Card Selected

Selected cards should be visibly distinct.

Allowed:

* stronger outline;
* selected label;
* subtle lift;
* stable highlight;
* type-consistent accent.

Selected state remains UI-local until the parent panel sends a command through `GameStateManager`.

### 7.3. Disabled / Denied Action

Denied action feedback must be clear.

Allowed:

* short shake;
* muted red pulse;
* disabled reason label highlight;
* button pulse.

Denied action must not mutate gameplay state.

Denied action must not fake success.

### 7.4. Purchase Success

After a successful purchase command:

* show quick purchase feedback;
* refresh view from `GameStateManager`;
* optionally pulse the bought card or market area;
* optionally highlight updated human board.

Do not show purchase success if facade result failed.

Do not compute price or affordability in feedback layer.

### 7.5. Attack Success

After a successful attack command:

* show quick impact feedback;
* optionally pulse target area;
* optionally highlight latest log entry.

Do not recompute combat result in UI.

Do not call `CombatEngine`.

### 7.6. Block / Defense Feedback

If the result/log/view indicates blocked attack:

* show shield/defense badge feedback;
* optionally pulse Cops/Cartel/Judge defense area if visible;
* use result/log/view data only.

UI must not calculate whether attack is blocked.

### 7.7. Discard Feedback

After a successful discard command:

* show card fade/slide or quick vanish feedback;
* refresh from facade state;
* do not directly remove card from gameplay hand.

### 7.8. Phase Transition

When phase changes:

* show short transition feedback;
* update phase header clearly;
* clear stale phase selection as already required by UI specs;
* keep core controls reachable.

Phase transition animation must not delay the actual committed state.

### 7.9. Street Deal Choice

After a successful Street Deal selection:

* pulse selected option;
* show short confirmation;
* refresh view.

Do not compute option effects in UI.

### 7.10. Contact Unlock / Select

After successful contact selection:

* pulse contact option or contact panel;
* clear stale pending offer through refreshed view.

Do not unlock contact directly.

### 7.11. Contract Claim

After successful contract claim:

* pulse claim button or contract panel;
* highlight updated reward state through refreshed view.

Do not compute reward.

### 7.12. Game Over Reveal

Game Over should have a presentation-only reveal.

Allowed:

* slow fade-in;
* result card/panel reveal;
* winner highlight;
* final score emphasis.

Forbidden:

* recomputing winner;
* changing final game result;
* delaying access to New Game for too long.

## 8. Timing Rules

Recommended timing ranges:

| Feedback              | Duration           |
| --------------------- | ------------------ |
| Hover                 | `0.05s` to `0.12s` |
| Selected              | `0.08s` to `0.15s` |
| Denied shake/pulse    | `0.15s` to `0.25s` |
| Purchase feedback     | `0.15s` to `0.35s` |
| Attack/block feedback | `0.15s` to `0.35s` |
| Discard feedback      | `0.15s` to `0.30s` |
| Phase transition      | `0.20s` to `0.45s` |
| Game Over reveal      | `0.50s` to `1.00s` |

Avoid long blocking animations.

No gameplay command should require waiting for a long animation to complete.

## 9. Presentation Lock Rules

### 9.1. Purpose

Presentation lock may prevent duplicate user input during a short UI feedback moment.

Allowed use cases:

* prevent double-clicking Buy;
* prevent double-clicking Execute Attack;
* prevent repeated End Market/End Action click;
* prevent repeated Contract Claim click;
* prevent repeated Street Deal option click.

### 9.2. Lock Ownership

Presentation lock must be:

* UI-local;
* short-lived;
* cleared after command result/refresh or after short feedback;
* independent from gameplay state.

### 9.3. Forbidden Lock Behavior

Presentation lock must not:

* write gameplay state;
* alter phase flow;
* block `GameStateManager` internally;
* hide failed command results;
* make replay depend on timing;
* create deadlock if animation is skipped in tests.

## 10. Scene Ownership

### 10.1. `GameRoot`

`GameRoot` may route facade signals to feedback layer.

Relevant facade signals:

* `state_changed`;
* `action_failed`;
* `phase_changed`;
* `game_started`;
* `game_ended`.

`GameRoot` must not:

* perform gameplay validation;
* mutate gameplay state;
* call owner modules directly;
* compute animation from hidden gameplay state.

### 10.2. `GameScreen`

`GameScreen` may:

* trigger phase transition feedback;
* trigger global error feedback;
* route panel-level feedback;
* clear UI selection on phase change.

`GameScreen` must not:

* compute combat/economy/contract effects;
* mutate player data;
* call owner modules directly.

### 10.3. `CardView`

`CardView` may own:

* hover feedback;
* selected feedback;
* disabled visual feedback;
* local card press feedback.

`CardView` must not:

* compute card price;
* compute card availability;
* parse effect text as rules;
* call owner modules;
* read `GameStateManager.state`.

### 10.4. `MarketPanel`

`MarketPanel` may own:

* purchase button feedback;
* selected market card feedback;
* denied purchase feedback;
* successful purchase visual response based on facade result.

`MarketPanel` must not compute purchase rules.

### 10.5. `ActionPanel`

`ActionPanel` may own:

* selected War card feedback;
* target/mode selection feedback;
* execute attack feedback;
* discard feedback;
* denied action feedback.

`ActionPanel` must not compute attack resolution.

### 10.6. `StreetDealPanel`

`StreetDealPanel` may own:

* option hover/selected feedback;
* successful choice feedback;
* denied choice feedback.

It must use canonical option IDs:

```text
option_a
option_b
```

### 10.7. `ContactPanel`

`ContactPanel` may own:

* contact selection feedback;
* pending offer feedback;
* denied selection feedback.

It must not unlock contacts directly.

### 10.8. `ContractPanel`

`ContractPanel` may own:

* claim button feedback;
* completed/claimable visual emphasis;
* denied claim feedback.

It must not calculate rewards.

### 10.9. `GameOverScreen`

`GameOverScreen` may own:

* result reveal;
* winner highlight;
* New Game button entrance.

It must not recompute winner.

* confirmed animation/feedback defect;
