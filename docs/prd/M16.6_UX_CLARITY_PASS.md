## 1. Purpose

M16.6 makes the existing polished UI easier to understand and play.

The player must always understand:

* what phase the game is in;
* whose turn it is;
* what the player can do now;
* what the player cannot do now;
* why a button/card/action is disabled;
* what happened after the last click;
* what changed after AI turns;
* what the next expected action is;
* how to proceed without reading implementation logs.

M16.6 exists to:

* reduce confusion in the current UI;
* improve phase-specific guidance;
* improve disabled reason readability;
* improve player-facing log summaries;
* improve next-action hints;
* clarify card/action/target selection;
* clarify AI activity and results;
* clarify contracts, contacts, Street Deals and debts;
* preserve `GameStateManager` as the only gameplay-facing UI API;
* preserve deterministic replay and all existing gameplay tests.

M16.6 must not implement new gameplay rules.

M16.6 must not add persistence, save/load, new game modes, campaign progression, balance changes or AI behavior changes.

| Next-action Hint | Short UI text explaining what the player should do next.                            |
| Disabled Reason  | Player-facing explanation for why a button/card/action cannot be used.              |

## 4. Runtime State

### 4.1. UX State Rule

M16.6 may store temporary UI-only clarity state.

Allowed UI-local clarity state:

* currently visible tooltip;
* currently expanded help section;
* current dismissed hint ID for the session;
* selected side panel tab;
* last visible result summary;
* last visible disabled reason;
* last highlighted next-action hint;
* local copy state for UI labels.

Forbidden UX gameplay state:

* player Nal;
* player VP;
* player hand contents as authority;
* market contents as authority;
* contract progress as authority;
* contact ownership as authority;
* debt state as authority;
* random step;
* phase state as authority;
* combat result as authority;
* AI decision state.

UX clarity must use:

```text
GameStateManager.get_view()
GameStateManager public selectors
safe presentation dictionaries
facade command results
facade signals
```

UX clarity must not read:

```text
GameStateManager.state
```

## 5. UX Clarity Goals

### 5.1. Immediate Understanding Rule

At any time during gameplay, the player should understand:

* current round;
* current phase;
* active player;
* human Nal/VP;
* current primary objective;
* available primary action;
* unavailable primary action reason;
* latest important outcome.

### 5.2. No Guessing Rule

The player should not need to guess:

* why a card cannot be bought;
* why an attack cannot be executed;
* why a contract cannot be claimed;
* why a contact cannot be selected;
* why Street Deal option is unavailable;
* what AI just did;
* whether the game is waiting for the player.

### 5.3. No Gameplay Explanation by Code IDs Rule

Raw technical IDs should not be the primary player-facing explanation.

Allowed:

* compact stable ID in debug tooltip if already available;
* raw ID as fallback for unknown errors;
* card ID in developer-only debug copy if not visually dominant.

Preferred:

```text
Not enough Nal.
Action already ended.
This card cannot target that player.
Contract is not complete yet.
Street Deal is available only in rounds 4, 8 and 12.
```

Not preferred as primary text:

```text
NOT_ENOUGH_NAL
INVALID_ATTACK_PAYLOAD
PHASE_NOT_READY
CONTRACT_NOT_COMPLETED
```

### 5.4. No False Promise Rule

Hints must not promise actions that are not actually available.

All actionable hints must be based on facade selectors, disabled reason checks, current view or command result.

UI must not invent availability logic.

## 6. Phase Clarity Rules

### 6.1. Setup Phase

Setup UI should clearly explain:

* choose seed;
* choose Turf Level;
* choose role;
* generate contract offers;
* choose one contract;
* start game.

Required clarity:

* Start disabled until required choices are valid;
* changing seed/role/Turf invalidates old contract offers;
* contract offer list shows that offers belong to current setup config.

### 6.2. Income Phase

Income phase should explain:

* income/upkeep has resolved or is ready to advance;
* what changed in important terms if visible from logs/view;
* next action is usually advancing to Market.

Do not recompute income in UI.

### 6.3. Market Phase

Market phase should explain:

* buy cards from market;
* rebuild district control if available;
* end market when done;
* AI will act after the human ends Market if facade orchestrates it.

Market card disabled reasons should explain:

* not enough Nal;
* card limit reached;
* phase/player not allowed;
* purchase not valid.

Do not compute price or affordability in UI.

### 6.4. Action Phase

Action phase should explain:

* select a War card;
* select target;
* select mode;
* select engine target if required;
* execute attack or discard;
* end action when done.

Disabled reasons should explain:

* no selected card;
* invalid target;
* invalid mode;
* missing engine target;
* attack blocked/unavailable;
* action already ended.

Do not compute combat rules in UI.

### 6.5. Street Deal Phase

Street Deal phase should explain:

* choose `option_a` or `option_b`;
* choice is human-only;
* some options may be unavailable;
* choice does not advance phase automatically unless current owner rules say so.

Player-facing labels must not show `A/B` as canonical payload IDs.

Canonical IDs:

```text
option_a
option_b
```

The UI may display them as “Option A” and “Option B” visually, but payloads must remain canonical.

### 6.6. Game Over Phase

Game Over should clearly display:

* winner;
* final scores;
* reason/result summary;
* New Game option.

Game Over UI must not recompute winner.

## 7. Panel Clarity Rules

### 7.1. Phase Header

Phase header must display:

* current round;
* current phase;
* active player;
* global error or last important result;
* short next-action hint.

The phase header should avoid long paragraphs.

### 7.2. PlayerBoard

PlayerBoard must clearly show:

* player identity;
* Nal;
* VP;
* important statuses;
* strong AI indicator if visible through view;
* enough compact info to understand board state.

PlayerBoard must not expose hidden AI internals.

### 7.3. MarketPanel

MarketPanel must clearly show:

* market cards;
* selected card;
* current/final price if supplied by facade/presentation layer;
* disabled reason near Buy;
* End Market button;
* Rebuild button if available.

If card price has scaling, the main displayed Market price should be the current final price supplied by facade/panel data, not merely base price.

CardView must not compute scaling.

### 7.4. ActionPanel

ActionPanel must clearly show:

* selected War card;
* selected target;
* selected attack mode;
* selected engine target if required;
* combat preview when available;
* Execute button;
* Discard button;
* End Action button;
* disabled reason near the relevant action.

### 7.5. StreetDealPanel

StreetDealPanel must clearly show:

* deal title/summary;
* both options;
* option availability;
* option disabled reason if available;
* current choice result or failure.

### 7.6. ContractPanel

ContractPanel must clearly show:

* current contract;
* objective/progress;
* deadline if relevant;
* claim state;
* disabled reason for claim.

Do not compute contract completion in UI.

### 7.7. ContactPanel

ContactPanel must clearly show:

* owned contact if any;
* pending offer if any;
* selection options;
* disabled reason for selection.

Do not unlock contact directly.

### 7.8. GameLogPanel

GameLogPanel should show readable summaries.

The log should be useful to a player, not only to a developer.

Preferred style:

```text
AI-1 bought Stash.
Human bought Laundry.
AI-2 attacked Human with Thug. Blocked by Cops.
Loan Shark debt was repaid.
Round 4 Street Deal offered.
```

Raw dictionaries should not be the primary visible log.

## 8. Disabled Reason Rules

### 8.1. Stable Error Mapping

All known validation errors used by UI should map to readable text.

Example mapping direction:

| Error Code                   | Player-facing Text                       |
| ---------------------------- | ---------------------------------------- |
| `OK`                         | empty / no error                         |
| `NOT_ENOUGH_NAL`             | Not enough Nal.                          |
| `PHASE_NOT_READY`            | Finish the current phase first.          |
| `INVALID_PLAYER_ID`          | Invalid player.                          |
| `PLAYER_ALREADY_MARKET_DONE` | This player already ended Market.        |
| `PLAYER_ALREADY_ACTION_DONE` | This player already ended Action.        |
| `INVALID_ATTACK_PAYLOAD`     | This attack is missing required choices. |
| `CONTRACT_NOT_COMPLETED`     | Contract is not complete yet.            |

Exact codes must be taken from existing `ValidationErrors`.

### 8.2. Unknown Error Fallback

Unknown error codes may be displayed literally as fallback.

Fallback must be clear enough:

```text
Unknown error: <code>
```

Do not hide unknown errors silently.

### 8.3. Locality Rule

Disabled reason should appear near the relevant control.

Examples:

* Buy disabled reason near Buy button;
* Execute disabled reason near Execute button;
* Contract claim reason near Claim button;
* Contact select reason near ContactPanel;
* global phase error in PhaseHeader.

## 9. Next-action Hint Rules

### 9.1. Hint Source Rule

Next-action hints must be derived from:

* current phase;
* current active player;
* facade view;
* allowed public selectors;
* disabled reason outputs.

Hints must not compute gameplay rules independently.

### 9.2. Hint Examples

Setup:

```text
Choose seed, role, Turf Level, then generate contract offers.
```

Income:

```text
Income resolved. Continue to Market.
```

Market:

```text
Buy cards or end Market.
```

Action:

```text
Select a War card and target, or end Action.
```

Street Deal:

```text
Choose one Street Deal option.
```

Game Over:

```text
Review final result or start a new game.
```

### 9.3. Hint Length Rule

Hints should be short.

Preferred:

```text
Buy cards or end Market.
```

Avoid:

```text
During the Market phase, the player may purchase cards according to the economy rules, taking into account price modifiers and current Nal...
```

## 10. Tooltip and Help Rules

### 10.1. Tooltip Scope

M16.6 may add tooltips for:

* card type;
* price;
* disabled button;
* phase header;
* contract progress;
* contact offer;
* defense badges;
* AI strong indicator;
* Nal/VP display.

Tooltips must be presentation-only.

### 10.2. Help Copy Scope

M16.6 may add minimal help copy:

* “What to do now” text;
* simple phase explanation;
* simple panel empty-state text.

M16.6 must not implement full tutorial/campaign onboarding unless explicitly approved.

### 10.3. No Spoiler / Hidden Info Rule

Tooltips and hints must not reveal hidden AI decision internals.

Allowed:

* AI profile visible name if already exposed by view;
* strong AI indicator if already exposed by view;
* visible board state.

Forbidden:

* future AI purchase plan;
* hidden scoring;
* internal random state;
* exact AI target formula.

## 11. Result Summary Rules

### 11.1. Command Result Summary

After a command, the UI should show a short result where practical.

Examples:

* Purchase succeeded.
* Attack executed.
* Attack blocked.
* Card discarded.
* Contract claimed.
* Street Deal selected.
* Contact unlocked.
* Action failed: Not enough Nal.

The source must be command result, facade signal, view/log update or safe selector.

### 11.2. AI Result Summary

After AI orchestration, UI should make visible that AI acted.

Allowed:

* latest log highlight;
* “AI turns completed” summary;
* compact recent action list if derived from log/view.

Forbidden:

* recomputing AI decisions in UI;
* summarizing hidden AI logic not present in view/log.

## 12. Copywriting Rules

### 12.1. Tone

Copy should be:

* short;
* direct;
* readable;
* criminal tabletop flavored where safe;
* not jokey;
* not overly modern dashboard language.

Examples:

Good:

```text
Not enough Nal.
Debt unpaid. Penalty applied.
Protection blocked the attack.
Claim reward.
End Market.
```

Avoid:

```text
Oopsie! You can't do that!
System error: purchase validator returned false.
According to the current business logic, this operation is not permitted.
```

### 12.2. Consistency

Use consistent terms:

* Nal;
* VP;
* Market;
* Action;
* Street Deal;
* Contract;
* Contact;
* Debt;
* War card;
* Engine;
* Status;
* Defense;
* AI.

Do not mix multiple names for the same concept unless localization explicitly supports it.

### 12.3. Localization Boundary

M16.6 may improve English player-facing copy.

Full localization system is out of scope unless already present.

Do not add multi-language architecture without approval.

## 17. Defect Fix Policy

* confirmed UX clarity defect;

* wrong/missing disabled reason display;

* panel empty-state defect;
