## 1. Purpose

M16.1 converts the already functional M16 UI from a panel/debug-like interface into a clearer desktop tabletop composition.

The UI should feel like a dark physical board/card table while preserving the existing gameplay architecture.

M16.1 exists to:

* improve the spatial hierarchy of the main game screen;
* make AI zones, human zone, market, action area and side information visually distinct;
* keep phase, round, active player and error feedback always visible;
* make the human player's core actions easy to find;
* keep market/action/Street Deal as the central gameplay area;
* keep contract/contact/log information visible but secondary;
* preserve `GameStateManager` as the only gameplay-facing UI API;
* preserve deterministic replay and all existing gameplay tests.

M16.1 must not implement new gameplay rules.

## 3. Core Terms

| Term               | Meaning                                                                                                                                                  |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Tabletop Layout    | Main game screen arranged like a physical card/board table instead of a debug panel stack.                                                               |
| Zone               | A spatial region of the game screen with a clear purpose, such as AI zone, Market zone, Human hand zone, or Side info zone.                              |
| Central Phase Area | Main interactive area that displays Market, Action, Street Deal, or phase-specific controls.                                                             |
| Human Zone         | Bottom or primary interaction area for the human player board, hand and action controls.                                                                 |
| AI Zone            | Compact board summary area for one AI participant.                                                                                                       |
| Side Info Column   | Secondary area for Contract, Contact and Game Log panels.                                                                                                |
| Layout State       | UI-only state such as expanded/collapsed sections, selected visible tab, or scroll position.                                                             |
| Gameplay State     | Authoritative runtime state owned by `GameStateManager`.                                                                                                 |
| Core Action        | Button-accessible gameplay action such as Buy, End Market, Execute Attack, Discard, End Action, Street Deal choice, Contact selection or Contract claim. |

## 4. Runtime State

### 4.1. UI Runtime State Rule

M16.1 may store temporary layout and presentation state only.

Allowed UI-local layout state:

* expanded player board;
* expanded log entry;
* selected side tab;
* currently visible phase panel;
* scroll position;
* hovered panel/card;
* selected card ID already allowed by `17_UI_UX_SPEC.md`;
* selected target/mode/option already allowed by `17_UI_UX_SPEC.md`.

Forbidden UI-local gameplay state:

* player Nal;
* player VP;
* player hand contents;
* market contents;
* contract progress;
* contact ownership;
* debt state;
* random step;
* phase state;
* combat result;
* AI decision state.

Gameplay state must come from:

```text
GameStateManager.get_view()
GameStateManager.get_state_snapshot()
GameStateManager public selectors
```

UI must not read:

```text
GameStateManager.state
```

## 5. Layout Rules

### 5.1. Desktop Layout Rule

The game targets desktop first.

Required desktop baseline:

```text
1280x720
```

Recommended target layout:

```text
1920x1080
```

The layout must support:

* landscape orientation;
* mouse input;
* readable text at 1080p;
* no mobile-first assumptions;
* no required drag-and-drop for core actions;
* button-based access to all core actions.

### 5.2. Tabletop Composition Rule

The main game screen should be arranged like a dark tabletop board.

Required high-level zones:

* phase header;
* AI zones;
* central phase area;
* human board/hand/action zone;
* side info column;
* error/disabled reason area.

Recommended structure:

```text
GameScreen / Tabletop Board

┌─────────────────────────────────────────────────────────────┐
│ Phase / Round / Active player / Error feedback              │
├───────────────┬───────────────────────────────┬─────────────┤
│ AI Zone       │ Market / Central Table         │ AI Zone     │
│ ai_1          │ active phase content           │ ai_2        │
├───────────────┴───────────────────────────────┴─────────────┤
│ AI Zone ai_3 / shared board summaries                       │
├───────────────────────────────┬─────────────────────────────┤
│ Human board + hand/action     │ Contract/Contact/Log column │
└───────────────────────────────┴─────────────────────────────┘
```

Exact node tree may differ if tests and usability remain correct.

### 5.3. Always Visible Information Rule

The following information should remain visible during normal gameplay:

* current phase;
* current round;
* active player;
* human Nal/VP summary;
* current error or disabled reason;
* main action buttons for the current phase.

At `1280x720`, secondary panels may scroll, but core controls must remain reachable.

### 5.4. Central Phase Area Rule

The central phase area owns phase-specific interaction.

It should display:

* Income continue control during Income;
* Market cards and purchase controls during Market;
* combat controls during Action;
* Street Deal offer during Street Deal;
* final status only when relevant.

The central phase area must not be occupied primarily by logs or secondary information.

### 5.5. Human Zone Rule

The human zone should be the clearest interaction area.

It should include:

* human board summary;
* human hand/cards when relevant;
* selected card/action controls;
* primary phase command button;
* local disabled reason near the relevant button.

The human player must not have to search across unrelated panels for the next valid action.

### 5.6. AI Zone Rule

Each AI must have a visible compact board summary.

AI zones should display:

* player ID or display name;
* Nal/VP;
* role/profile/strong AI indicator if available;
* compact card/defense/status summary;
* recent action indicator if available through existing logs/view data.

AI zones must not expose hidden AI decision internals.

### 5.7. Side Info Column Rule

Contract, Contact and Game Log belong to secondary information space.

The side info column may contain:

* ContractPanel;
* ContactPanel;
* GameLogPanel;
* optional compact tabs or collapsible groups.

Side info must not block:

* Buy;
* End Market;
* Execute Attack;
* Discard;
* End Action;
* Street Deal choice;
* Contact selection;
* Contract claim.

### 5.8. Scroll Rule

ScrollContainer may be used for:

* long logs;
* side info;
* oversized player board details;
* low-resolution fallback.

ScrollContainer must not hide all core phase actions at once.

At `1280x720`, the layout may scroll secondary content, but the phase header and core current-phase controls must remain reachable.

### 5.9. No Debug Stack Rule

The Game screen must not remain a plain vertical stack of unrelated panels.

Forbidden final layout smell:

* all panels stacked vertically without spatial hierarchy;
* log dominating the primary screen;
* market/action controls far away from selected card;
* player boards hidden behind excessive scrolling at `1920x1080`;
* default prototype composition with no clear table zones.

## 6. Scene Ownership

### 6.1. `Main`

`Main` owns:

* project entry scene;
* root theme application if already established;
* initial `GameRoot` placement.

`Main` must not:

* own gameplay state;
* call owner gameplay modules;
* perform phase orchestration.

### 6.2. `GameRoot`

`GameRoot` owns:

* screen switching;
* signal connection to `GameStateManager`;
* routing refreshed views to screens;
* active screen visibility.

`GameRoot` must not:

* duplicate panel logic;
* mutate gameplay state;
* perform gameplay validation;
* call owner modules directly.

### 6.3. `GameScreen`

`GameScreen` owns:

* tabletop layout composition;
* phase header placement;
* AI zone placement;
* central phase area placement;
* human zone placement;
* side info placement;
* layout-level refresh routing.

`GameScreen` may clear UI-local selection on phase changes.

`GameScreen` must not:

* compute gameplay results;
* change phase fields;
* mutate player data;
* run AI logic directly;
* parse card effects.

### 6.4. `SetupScreen`

`SetupScreen` owns:

* seed input;
* Turf Level selection;
* role selection;
* contract offer display;
* selected setup contract.

If seed, role or Turf Level changes after contract offers were generated, the UI must invalidate stale contract offers.

Required behavior:

* clear selected contract;
* reset old contract options;
* disable Start until offers are regenerated and a valid offer is selected.

### 6.5. `GameOverScreen`

`GameOverScreen` owns:

* winner display;
* final result display;
* New Game control.

It must not:

* allow gameplay mutations after Game Over except reset/new game through facade;
* recompute winner;
* alter final state.

## 7. Panel Placement Rules

### 7.1. `PlayerBoard`

PlayerBoard must support compact display.

Required display:

* player identity;
* Nal/VP;
* relevant card summaries;
* defense/status/engine summary where view data exists.

PlayerBoard must not mutate gameplay state.

### 7.2. `MarketPanel`

MarketPanel should appear in the central phase area during Market.

Required behavior:

* display market cards;
* display selected card preview;
* show price and disabled reason from facade selectors;
* expose Buy/Rebuild/End Market buttons where relevant.

MarketPanel must not compute price or affordability.

### 7.3. `ActionPanel`

ActionPanel should appear in the human/action zone or central phase area during Action.

Required behavior:

* display human War cards/hand interaction where available;
* display target/mode/engine target selectors;
* display combat preview;
* expose Execute, Discard and End Action buttons.

ActionPanel must not resolve combat or calculate block effects.

### 7.4. `StreetDealPanel`

StreetDealPanel should appear prominently during Street Deal phase.

Required behavior:

* display current deal;
* display `option_a` and `option_b`;
* expose option selection;
* show disabled reason where available.

StreetDealPanel must use canonical option IDs:

```text
option_a
option_b
```

It must not use:

```text
A
B
```

### 7.5. `ContractPanel`

ContractPanel belongs to the side info column.

Required behavior:

* display current human contract;
* display progress/completion/claim state from view/facade;
* expose Claim button when appropriate.

ContractPanel must not compute contract completion.

### 7.6. `ContactPanel`

ContactPanel belongs to the side info column.

Required behavior:

* display owned contact state;
* display pending contact offer if available;
* expose contact selection through facade.

ContactPanel must not unlock contacts directly.

### 7.7. `GameLogPanel`

GameLogPanel belongs to the side info column.

Required behavior:

* display readable log summaries;
* allow expanded detail if already supported;
* remain scrollable when long.

GameLogPanel must not create gameplay log entries.

## 9. Known M16 Defects Covered by M16.1

M16.1 may fix confirmed UI-local defects if still present.

### 9.1. Stale Setup Contract Offers

Defect:

* user generates contract offers;
* user selects a contract;
* user changes seed, role or Turf Level;
* old contract options remain selectable.

Required fix:

* invalidate offers when seed/role/turf changes;
* clear selected contract;
* reset contract options display;
* disable Start until offers are regenerated.

This is a UI-local state fix and does not change gameplay setup logic.

### 9.2. UI Static Boundary Root Coverage

Static scans and Cursor hooks should treat all of these as UI roots:

```text
res://scenes/main
res://scenes/game
res://scenes/ui
```

UI static scans must prevent:

* direct gameplay state mutation;
* owner module calls;
* gameplay random calls;
* UI file length violations.
